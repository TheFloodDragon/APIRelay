# APIRelay 改进实施计划

> 来源：2026-07-09 全库深度审计（多智能体 7 维并行审查 + 独立核查，约 70 条候选结论，收录前逐条对照源码复核）。
> 原则：不考虑向后兼容，允许破坏性重构；每项改动须附验收标准；按批次推进，单批可独立发布。
> 状态图例：☐ 待办 ◐ 进行中 ☑ 完成 ✕ 放弃

---

## 一、背景与目标

APIRelay 的架构骨架（IR 中枢、Adaptor/Outbound 对称解耦、故障转移状态机、预扣/结算/退款计费会话、滑动窗口熔断）是健全的。审计暴露的问题集中在三类：

1. **转发核心正确性**：错误分类误判、跨协议流式工具调用损坏、流中断伪装成功等，会真实吞错误、断流、丢数据；
2. **「零改写透传」只做了一半**：请求侧全部经有损 IR 重建，`tool_choice`/`thinking`/`cache_control` 等字段被静默丢弃，同协议直转也不例外；
3. **运行时短板**：无优雅停机（重启丢账）、SQLite 零调优（并发丢写）、冷却与熔断双轨冲突、热路径无缓存。

目标：分三批把上述问题清零，同时完成管理台 UI 全翻新（独立工作流，见第四节）。

## 三、剩余工作分批实施

> 2026-07-10 全量源码复核后重新排序。已完成项保留在第二节作为历史证据，不再进入后续批次。

### 第一批：转发正确性止血（P0）

顺序：A4 → A11 → A10。

- **A4 流中断语义**：仅在上游自然完成时写协议结束事件；中断后保留截断语义，并按已产生的 usage/估算量结算而非无条件全额退款。
- **A11 重试边界**：把“是否已向客户端写出内容”纳入故障转移决策；任何已写流的 timeout/网络错误均不可启动第二个上游。
- **A10 模型名回写**：同协议 Raw SSE 透传时，只定点改写响应中的顶层模型字段为 `OriginModel`，不破坏未知字段和事件边界。

验收标准：
- mock 上游首包后主动断连，客户端不得收到 `[DONE]`/`message_stop`，不得拼接第二渠道响应；日志明确标记中断；
- 首包前 timeout 仍可切换渠道，首包后 timeout 不切换；
- OpenAI、Anthropic、Responses 同协议映射场景均只向客户端暴露显示模型名。

### 第二批：协议与计费保真（P1）

顺序：B2 → B4 → B3 → F4。

- **B2**：补齐 `ToolChoice`、`ResponseFormat`、`Thinking`/signature、`TopK`，复用已完成的多模态 Parts 与工具调用 Index。
- **B4**：建立端点能力矩阵；跨协议无法表达的字段在发起上游请求前返回 400，不再静默降级。
- **B3**：扩展 Usage 和日志字段，支持 cache creation/read、reasoning tokens 及可配置倍率计费。
- **F4**：仅在上游不返回 usage 时启用 completion token 估算，并在日志中标注 estimated。

验收标准：
- 三协议两两转换覆盖字段保留/拒绝矩阵；
- thinking、tool choice、response format 与多模态组合请求有 e2e；
- cache read/write/reasoning 用量解析、倍率计费、预扣与结算均有精确断言；
- 实际 usage 优先于估算，估算路径不会覆盖真实值。

### 第三批：可靠性与状态一致性（P0/P1）

顺序：C3+D5 → C5 → C6+C7 → C8 → C4 → C9。

- **C3+D5**：删除热路径 cooldown 设置/清理，渠道摘除统一由熔断器负责。
- **C5**：熔断配置统一以 settings 为运行时单源；首次启动可从 YAML 初始化，后续启动必须加载持久化配置。
- **C6+C7**：熔断快照增加单调序号并串行持久化；删除渠道时同步删除 breaker 实例和 `channel_health`。
- **C8**：解析 `Retry-After`，明确同渠道重试锁定语义，并区分网关自身 timeout 与上游失败。
- **C4**：拆分首字节超时、流空闲超时和完整请求上限；空闲看门狗必须可被 context 取消。
- **C9**：滑动窗口改增量计数或环形桶，消除每请求锁内 O(n) 重扫。

验收标准：
- 单渠道一次 502 后仍按熔断阈值决定是否可用，不再固定冷却 60 秒；
- 重启后管理台保存的熔断配置保持生效；Reset/删除与并发快照不会复活旧状态；
- 429 尊重秒值和 HTTP-date 两种 `Retry-After`；
- 上游停流可在 idle timeout 内退出，正常长流不会被首字节超时误杀；
- 熔断窗口基准测试验证锁持有时间和分配下降。

### 第四批：热路径性能（P1）

顺序：D2 → D1 → D3 → D4。

- **D2**：渠道加载时解析一次 `ModelConfigs`/mapping/rules，形成不可变运行时快照。
- **D1**：对候选渠道/Ability 建进程内快照并由渠道事务提交后失效；Token 使用短 TTL 缓存，禁用、充值、删改时立即失效。
- **D3**：Raw SSE 先按 event/usage 特征预筛，再按事件边界 flush，避免每行完整反序列化和重复 flush。
- **D4**：在现有 system/manual/direct 代理支持基础上启用 HTTP/2，并按压测调整连接池；保留 Windows 系统代理和 Fake-IP 防护语义。

验收标准：
- 候选查询基准 QPS 提升 ≥ 5×，缓存失效后无陈旧渠道/令牌；
- `go test -race ./...` 覆盖缓存并发读写；
- 1000 SSE 事件基准记录 allocations、解析次数和 flush 次数；
- 代理三模式、HTTP/2 与连接复用均有测试。

### 第五批：安全与管理能力（P2）

顺序：E3 → E4 → F1 → F2 → F3 → E5。

- **E3**：渠道列表只返回脱敏 key；新增受管理员会话保护的按需详情读取，写入时允许留空表示保持原 key。
- **E4**：新增 `PUT /api/auth/password`，校验旧密码、更新哈希并吊销该用户现有会话。
- **F1**：新增 Token 编辑/禁用/启用、增量充值、模型白名单、分组和过期时间修改；所有变更同步失效鉴权缓存。
- **F2**：日志保留期清理、索引、request-id 精确/前缀搜索和 keyset 分页；保留兼容分页需明确上限。
- **F3**：无价格模型记录 WARN；`strict_pricing` 可拒绝未配置价格的请求；管理台展示未配价差集。
- **E5**：提供默认关闭的上游 host/CIDR allowlist 与按 Token 限速，避免改变自部署默认行为。

验收标准：
- 列表与日志中不出现完整渠道 key；改密后旧会话立即失效；
- Token 充值使用数据库原子增量，禁用后下一请求立即被拒绝；
- 百万级日志数据下分页和清理有查询计划/耗时基准；
- strict pricing、allowlist 和限速均覆盖关闭/开启两种配置。

### 第六批：工程化与收尾（P2）

顺序：G1 → G6 剩余项 → F5 → F6。

- **G1**：`/assets/` 使用 immutable 长缓存且未命中返回 404；`index.html` no-cache；SPA 路由只对无扩展名页面路径 fallback。
- **G6**：版本号收敛为单一构建来源。
- **F5**：CI 增加 `-race`、golangci-lint、Docker 镜像构建与发布；补本地统一命令入口。
- **F6**：按 handler/failover/stream/billing hook 拆分 `relayer.go`，收敛协议枚举并删除确认无引用的 Go 死代码。

验收标准：
- 静态资源缓存头、404 和 SPA fallback 有 HTTP 测试；
- CI 在 Linux 执行 race/lint，镜像可启动并通过 `/healthz`；
- 重构前后现有测试与协议 e2e 全绿，核心文件职责和依赖方向清晰。

**设计方向**：「工程图纸 / 电网调度」。依据：中转站的领域词汇本就是电气的——继电器（relay）、熔断器（circuit breaker）、冷却停运；渠道即馈线、令牌即用电户、日志即事件顺序记录（SOE）。视觉：白底工程制图（细线框、图签 title block、DIN 风格标注字、等宽数字），熔断状态用 IEC 断路器符号语义（闭合/半开/断开）。签名元素：总览页的**单线图**（客户端 → 母线 → 各渠道馈线，SVG 实时反映熔断状态）。单一浅色主题（有意决策，删除失效的双主题死代码）。

**范围**：
