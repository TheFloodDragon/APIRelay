# APIRelay 改进实施计划

> 来源：2026-07-09 全库深度审计（多智能体 7 维并行审查 + 独立核查，约 70 条候选结论，收录前逐条对照源码复核）。
> 原则：不考虑向后兼容，允许破坏性重构；每项改动须附验收标准；按批次推进，单批可独立发布。
> 状态图例：☐ 待办 ◐ 进行中 ☑ 完成 ✕ 放弃

---

## 一、背景与目标

APIRelay 的架构骨架（IR 中枢、Adaptor/Outbound 对称解耦、故障转移状态机、预扣/结算/退款计费会话、滑动窗口熔断）是健全的。审计暴露的问题集中在三类：

1. **转发核心正确性**：错误分类误判、跨协议流式工具调用损坏、流中断伪装成功等，会真实吞错误、断流、丢数据；
2. **「零改写透传」只做了一半**：请求侧全部经有损 IR 重建，`tool_choice`/`thinking`/`cache_control` 等字段被静默丢弃，同协议直转也不例外；
3. **运行时短板**：无优雅停机（重启丢账）、SQLite 零调优（并发丢写）、冷却与熔断双轨冲突、热路径无缓存。

目标：分三批把上述问题清零，同时完成管理台 UI 全翻新（独立工作流，见第四节）。

## 三、分批实施

### 第一批（P0 止血，每项 ≤ 半天）

顺序：A1 → C1 → C2 → A3 → A5 → A8 → A7+A9 → E1。

验收标准（关键项）：
- A1：表驱动测试覆盖错误分类矩阵（client cancel / relay timeout / upstream timeout / conn reset / broken pipe / EOF），`connection reset` 必须归 upstream 且触发转移；
- C1：`kill -TERM` 后进程等待 in-flight 完成（上限 30s）、async 队列排空、日志 flush，`used_quota` 无残留预扣（e2e 断言）；
- C2：并发 50 请求压测下无 `SQLITE_BUSY` 丢结算（日志断言 settle 无 error）；
- A3/A5：mock 上游返回 401/首包前 502，断言换渠道重试且客户端最终拿到正确状态码。

### 第二批（P1 核心重构，1~2 周）

顺序：B1 → B2 → A2+A6 → B3 → C3 → D1+D2 → C4 → C5 → D3+D4。

验收标准（关键项）：
- B1：同协议 e2e——发送含 `cache_control`/`tool_choice`/`thinking` 的 Anthropic 请求，mock 上游收到的 body 与原请求仅 `model` 字段不同（字节级 diff）；
- A2：OpenAI↔Anthropic 双向流式多工具调用 e2e，客户端解析出的工具调用数量、id、参数与上游一致；
- B3：带 `cache_read_input_tokens` 的 usage 计费断言（0.1× 输入价）；
- C3：删除 `SetChannelCooldown` 后，单渠道 + 一次 502 场景下第二个请求仍可尝试该渠道（由熔断阈值决定摘除）；
- D1：压测对比：候选查询 QPS 提升 ≥ 5×，无功能回归（现有测试全绿）。

### 第三批（P2 打磨）

F1 → F2 → E3+E4 → F3+F4 → G1 → F5 → F6 → C6~C9 → E5。

## 四、UI 全翻新工作流（本次实施 ◐）

**决策**：完全重写 `web/src`，不沿用 patchbay 视觉体系的任何元素（组件、配色、隐喻全部废弃）。

**设计方向**：「工程图纸 / 电网调度」。依据：中转站的领域词汇本就是电气的——继电器（relay）、熔断器（circuit breaker）、冷却停运；渠道即馈线、令牌即用电户、日志即事件顺序记录（SOE）。视觉：白底工程制图（细线框、图签 title block、DIN 风格标注字、等宽数字），熔断状态用 IEC 断路器符号语义（闭合/半开/断开）。签名元素：总览页的**单线图**（客户端 → 母线 → 各渠道馈线，SVG 实时反映熔断状态）。单一浅色主题（有意决策，删除失效的双主题死代码）。

**范围**：
- 重写：`tailwind.config.js`、`style.css`（设计令牌）、`index.html`、`App.vue`（外壳：图纸边框 + 目录导航 + 状态栏）、7 个视图、全部组件（Toast 重做，废弃 MeterBar/PatchRoute/PriorityBar/ProtocolTag/SignalDot/StatDial）、字体（弃 IBM Plex，换 Saira + 等宽数据字体）；
- 保留但增强：`api.js`（加请求竞态守卫与登出吊销）、`router.js`（路由不变）；
- 功能对等：以《现有前端功能规格》为核对清单，渠道编辑器（model_configs/协议规则/header_override/探测/测试）、日志筛选全集、设置四区块一项不落；
- 顺带修复：G2~G6 全部。

**验收**：`npm run build` + `go build` 通过；逐页截图检查（登录/总览/渠道/模型/令牌/日志/设置）；键盘可达（焦点可见、Esc 关模态）；≤1024px 可用（操作不隐藏）；`prefers-reduced-motion` 生效。

## 五、测试与验收策略

1. **错误分类矩阵表驱动测试**（A1/A3/A5/A11 的回归防线）；
2. **协议保真 e2e**：testmock 上游断言收到的请求体（B1 字节级 diff）；跨协议工具调用双向用例（A2）；
3. **计费一致性**：预扣→结算→退款在成功/中断/崩溃三路径的 `used_quota` 断言（C1/A4/B3）；
4. **压测脚本**：50 并发 × 1min，断言零 BUSY 丢写、零 goroutine 泄漏（C2/C4/D1）；
5. CI 补 `-race` 与 golangci-lint（F5）。
