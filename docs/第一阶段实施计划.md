# APIRelay 第一阶段实施计划：安全与资源保护

> 生成日期：2026-07-05  
> 范围：第一阶段只处理“启动安全、管理会话、登录防护、CORS、请求体限制、上游请求 context/timeout、前端登录页提示”。计费一致性、熔断滑动窗口、显式 failover 队列、前端设计系统重构放到后续阶段。

## 1. 阶段目标

第一阶段的目标是先降低线上暴露风险和资源耗尽风险，让 APIRelay 在进入更复杂的计费、熔断和策略重构之前具备安全可控的运行基线。

核心目标：

1. 首次启动默认兼容创建 `admin/admin123`，但必须写 warning 且生产可关闭。
2. 日志不输出完整管理员密码、root token、API key、session token。
3. 管理后台登录具备基础防暴力破解能力。
4. 管理 session 使用 `auth.session_secret`，避免当前配置项闲置。
5. `/v1/*` 与 `/api/*` 都有请求体大小限制。
6. relay 上游请求绑定客户端 context 和可配置 timeout。
7. 客户端取消、上游超时、上游失败在日志与熔断上有不同处理。
8. CORS 从硬编码 `*` 改为可配置 allowlist，并补齐必要 header。
9. 前端登录页不在生产环境展示默认账号密码。

## 2. 当前代码事实

### 2.1 启动与 bootstrap

涉及文件：

- `main.go`
- `common/config/config.go`
- `config.example.yaml`
- `README.md`

当前行为：

- `bootstrap(cfg)` 在用户数为 0 时固定创建：
  - username: `admin`
  - password: `admin123`
- 初始管理员密码通过 `logger.L().Info(... zap.String("password", pw))` 明文打印。
- root token 为空时自动生成，并通过 `zap.String("key", plain)` 明文打印。
- 非首次启动时，如果配置了 `auth.initial_root_token`，也会在 ensured 日志中打印完整 token。

风险：

- 容器 stdout、日志采集系统、终端历史或面板日志泄露即可接管系统。
- 默认密码如果没有修改，任何可访问管理后台的人都可登录。

### 2.2 管理后台登录与 session

涉及文件：

- `controller/auth.go`
- `common/config/config.go`

当前行为：

- 登录成功后生成 `sess-` 前缀随机 token。
- session 只存在进程内 `map[string]session`。
- `auth.session_secret` 配置存在，但未用于签名或校验。
- 无登录失败限速、无账号锁定、无失败审计。

风险：

- 可暴力破解默认密码。
- 多实例部署 session 不共享。
- 重启后全部失效；这本身不一定是安全问题，但需要明确设计。

### 2.3 路由、中间件与 CORS

涉及文件：

- `router/router.go`
- `middleware/common.go`
- `middleware/auth.go`

当前行为：

- `middleware.CORS()` 对所有请求设置 `Access-Control-Allow-Origin: *`。
- Allow-Headers 只有 `Origin, Content-Type, Authorization, X-Request-Id`。
- 未包含 `x-api-key`、`X-Session-Token`。
- `/v1/*` 和 `/api/*` 没有请求体大小限制。

风险：

- 生产环境跨域策略过宽。
- 浏览器侧使用 `x-api-key` 或 `X-Session-Token` 可能预检失败。
- 大 body 可造成内存压力。

### 2.4 Relay 请求生命周期

涉及文件：

- `relay/relayer.go`
- `relay/relaycommon/relayinfo.go`
- `relay/adaptor/adaptor.go`
- `relay/adaptor/openai/adaptor.go`
- `relay/adaptor/anthropic/adaptor.go`
- `relay/adaptor/responses/adaptor.go`
- `relay/adaptor/client.go`

当前行为：

- `relay/relayer.go` 使用 `io.ReadAll(c.Request.Body)`，无大小限制。
- `config.RelayConfig.RequestTimeout` 存在，但 relay 主链路没有真正创建 timeout context。
- adaptor 使用 `http.NewRequest(...)`，不是 `http.NewRequestWithContext(...)`。
- `adaptor.HTTPClient().Timeout` 为 0，注释称“由 context 控制”，但实际 context 未传入。
- `relay/relayer.go` 目前有 `var _ = context.Background` 占位，说明 context 尚未真正接入。
- 流式 `writer.WriteChunk` 返回错误会被包装成 `stream: ...`，之后上层会记录熔断失败，客户端断开可能误伤上游健康。

风险：

- 客户端断开后，上游请求可能继续消耗连接、token 和 goroutine。
- 非流式上游长时间卡住时没有请求级超时保护。
- 客户端取消与上游失败混淆，可能导致错误熔断。

### 2.5 前端登录页

涉及文件：

- `web/src/views/Login.vue`

当前行为：

- username 默认填充 `admin`。
- 页面底部展示：`默认账号：admin / admin123`。

风险：

- 生产环境给攻击者提供明确猜测目标。
- 与后端默认密码问题叠加，风险更高。

## 4. 配置设计

### 4.1 建议新增配置字段

在 `common/config/config.go` 中扩展：

```go
type ServerConfig struct {
    Port int `yaml:"port"`
    Host string `yaml:"host"`

    // AdminMaxBodyBytes 管理 API 请求体上限，默认 2 MiB。
    AdminMaxBodyBytes int64 `yaml:"admin_max_body_bytes"`

    // CORSAllowedOrigins 生产环境建议配置为控制台域名；为空时仅允许同源/非浏览器请求。
    CORSAllowedOrigins []string `yaml:"cors_allowed_origins"`
}

type RelayConfig struct {
    MaxRetries int `yaml:"max_retries"`
    ChannelMaxRetries int `yaml:"channel_max_retries"`
    CooldownSeconds int `yaml:"cooldown_seconds"`
    RequestTimeout int `yaml:"request_timeout"`

    // MaxBodyBytes Relay API 请求体上限，默认 20 MiB。
    MaxBodyBytes int64 `yaml:"max_body_bytes"`

    DefaultGroup string `yaml:"default_group"`
    CircuitBreaker CircuitBreakerConfig `yaml:"circuit_breaker"`
}

type AuthConfig struct {
    SessionSecret string `yaml:"session_secret"`
    InitialRootToken string `yaml:"initial_root_token"`

    // InitialAdminUsername 首启管理员用户名，默认 admin。
    InitialAdminUsername string `yaml:"initial_admin_username"`

    // InitialAdminPassword 首启管理员密码。生产建议必须显式配置。
    InitialAdminPassword string `yaml:"initial_admin_password"`

    // AllowInsecureDefaultAdmin 是否允许首启回退 admin/admin123。
    // 默认 true；兼容旧登录体验，生产建议关闭。
    AllowInsecureDefaultAdmin bool `yaml:"allow_insecure_default_admin"`

    // LoginMaxFailures 同一 IP + username 在窗口内允许失败次数，默认 5。
    LoginMaxFailures int `yaml:"login_max_failures"`

    // LoginFailureWindowSeconds 失败计数窗口，默认 600 秒。
    LoginFailureWindowSeconds int `yaml:"login_failure_window_seconds"`

    // LoginLockoutSeconds 超过失败次数后的锁定时间，默认 900 秒。
    LoginLockoutSeconds int `yaml:"login_lockout_seconds"`
}
```

环境变量建议：

- `APIRELAY_INITIAL_ADMIN_USERNAME`
- `APIRELAY_INITIAL_ADMIN_PASSWORD`
- `APIRELAY_ALLOW_INSECURE_DEFAULT_ADMIN`
- `APIRELAY_SESSION_SECRET`
- `APIRELAY_INITIAL_ROOT_TOKEN`
- `APIRELAY_RELAY_MAX_BODY_BYTES`
- `APIRELAY_ADMIN_MAX_BODY_BYTES`
- `APIRELAY_CORS_ALLOWED_ORIGINS`，逗号分隔
- `APIRELAY_REQUEST_TIMEOUT`
- `APIRELAY_LOGIN_MAX_FAILURES`
- `APIRELAY_LOGIN_FAILURE_WINDOW_SECONDS`
- `APIRELAY_LOGIN_LOCKOUT_SECONDS`

### 4.2 默认值建议

```text
server.admin_max_body_bytes = 2097152       # 2 MiB
server.cors_allowed_origins = []            # 不再默认通配所有 origin
relay.max_body_bytes = 20971520             # 20 MiB
relay.request_timeout = 0                   # 保持兼容，0 表示不限

auth.initial_admin_username = "admin"
auth.initial_admin_password = ""
auth.allow_insecure_default_admin = true
auth.login_max_failures = 5
auth.login_failure_window_seconds = 600
auth.login_lockout_seconds = 900
```

兼容策略：

- 按当前兼容要求，首启无用户且未配置 `initial_admin_password` 时默认允许回退 `admin/admin123`。
- 该旧行为必须打 `WARN` 日志，且不再打印完整密码。
- 生产环境建议显式配置：
  - YAML: `auth.initial_admin_password` 与 `auth.session_secret`
  - YAML: `auth.allow_insecure_default_admin: false`
  - 或对应环境变量覆盖。

## 5. 实施任务分解

### 5.1 任务 A：配置扩展与校验

涉及文件：

- `common/config/config.go`
- `config.example.yaml`
- `README.md`

改造步骤：

1. 扩展 `ServerConfig`、`RelayConfig`、`AuthConfig`。
2. 更新 `Default()` 默认值。
3. 扩展 `applyEnv()` 环境变量覆盖。
4. 增加配置规范化函数，例如：

```go
func (c *Config) Normalize()
func (c *Config) ValidateStartup() error
```

5. 对关键字段做下限保护：
   - body limit 小于等于 0 时使用默认值，除非明确设计为 0 表示不限。
   - 登录失败次数小于等于 0 时使用默认值。
   - session secret 为空时在生产安全模式下拒绝启动。

验收：

- `config.example.yaml` 包含新增字段与注释。
- 环境变量能覆盖新增字段。
- 错误配置在启动时给出明确错误。

### 5.2 任务 B：安全 bootstrap

涉及文件：

- `main.go`
- `common/config/config.go`
- 可选新增：`common/secret/secret.go` 或 `common/mask.go`

改造步骤：

1. `bootstrap(cfg)` 改为返回 error，由 `main()` 决定 fatal 退出。
2. 首启 `CountUsers() == 0` 时：
   - username 使用 `cfg.Auth.InitialAdminUsername`，默认 `admin`。
   - password 优先使用 `cfg.Auth.InitialAdminPassword`。
   - 若 password 为空且 `AllowInsecureDefaultAdmin=false`，启动失败并提示设置 `APIRELAY_INITIAL_ADMIN_PASSWORD`。
   - 若 password 为空且 `AllowInsecureDefaultAdmin=true`，才使用 `admin123`，并写 `WARN`。
3. root token：
   - 如果 `cfg.Auth.InitialRootToken` 为空，可以继续自动生成。
   - 日志只打印脱敏结果，例如 `sk-abcd...wxyz`。
   - README 需说明：生产环境建议显式配置并安全保存。
4. 非首次启动 ensure root token 时，也只打印脱敏 token。

建议新增工具：

```go
func MaskSecret(s string) string
```

规则：

- 长度 <= 8：返回 `***`。
- 长度 > 8：返回前 4 + `...` + 后 4。

验收：

- 默认首次启动在无 admin password 时失败。
- 显式配置 `APIRELAY_INITIAL_ADMIN_PASSWORD` 后可创建管理员。
- 日志中不出现完整 password/root token。
- `allow_insecure_default_admin=true` 时仍可兼容旧本地体验，但日志有明显 warning。

### 5.3 任务 C：管理 session 使用 SessionSecret

涉及文件：

- `controller/auth.go`
- `common/config/config.go`
- `router/router.go`

改造策略：

第一阶段采用轻量实现，不引入完整 session 存储系统：

1. 增加 `controller.InitAuth(cfg.Auth)` 或把 auth manager 注入 router。
2. 使用 HMAC-SHA256 签名管理 session token。
3. token payload 包含：
   - `user_id`
   - `username`
   - `exp`
   - `jti`
4. token 形态：

```text
sess-<base64url(payload)>.<base64url(signature)>
```

5. `AdminAuth()` 验证签名和过期时间。
6. `AdminLogout()` 第一阶段可维护内存 revoke list：
   - key: `jti`
   - value: expiresAt
   - 定期或请求时清理过期项。
7. 保持 24 小时 TTL，后续阶段再做 refresh/rotation。

兼容策略：

- 不兼容旧内存 session 是可以接受的；升级后用户重新登录。
- 如果 `session_secret` 为空：
  - 安全模式下启动失败。
  - 本地 insecure 模式下可生成临时 secret，但重启会使 session 失效，并写 warning。

验收：

- `auth.session_secret` 真正参与 session 签名。
- 修改 secret 后旧 session 失效。
- logout 后当前 token 失效。
- 过期 token 返回 `会话已过期`。

### 5.4 任务 D：登录限速与失败审计

涉及文件：

- `controller/auth.go`
- 可选新增：`controller/login_limiter.go`

改造步骤：

1. 新增内存登录限速器，key 为：

```text
client_ip + "\x00" + username
```

2. 记录字段：
   - `failures`
   - `windowStartedAt`
   - `lockedUntil`
3. 登录流程：
   - bind request 后先检查是否 locked。
   - 登录失败记录 failure。
   - 达到阈值后锁定。
   - 登录成功清除该 key 的失败记录。
4. 日志：
   - 登录失败写 warn，包含 username、ip、是否 locked。
   - 不记录密码。

验收：

- 同 IP + username 连续失败超过阈值后返回 429 或 403。
- 锁定期结束后可以再次尝试。
- 登录成功清空失败计数。

### 5.5 任务 E：请求体大小限制

涉及文件：

- `middleware/common.go`
- `router/router.go`
- `relay/relayer.go`
- 各 controller bind JSON 的错误处理保持兼容

改造步骤：

1. 新增中间件：

```go
func BodySizeLimit(maxBytes int64) gin.HandlerFunc
```

内部使用：

```go
c.Request.Body = http.MaxBytesReader(c.Writer, c.Request.Body, maxBytes)
```

2. `/v1/*` 路由组添加：

```go
v1.Use(middleware.BodySizeLimit(cfg.Relay.MaxBodyBytes))
```

3. `/api/*` 受保护路由组和 `/api/auth/login` 添加：

```go
middleware.BodySizeLimit(cfg.Server.AdminMaxBodyBytes)
```

4. `relay.handle()` 中 `io.ReadAll` 如果遇到 `*http.MaxBytesError`，返回 413。
5. 管理 API 中 `ShouldBindJSON` 遇到 body too large 时返回统一错误。

验收：

- 超出 relay body limit 返回 413。
- 超出 admin body limit 返回 413。
- 正常请求不受影响。

### 5.6 任务 F：CORS 可配置化

涉及文件：

- `middleware/common.go`
- `router/router.go`
- `common/config/config.go`
- `config.example.yaml`

改造步骤：

1. `middleware.CORS()` 改为：

```go
func CORS(allowedOrigins []string) gin.HandlerFunc
```

2. 若 `allowedOrigins` 包含 `*`，保留通配行为，但配置和示例中不推荐。
3. 若为空：
   - 对没有 `Origin` 的非浏览器请求不设置 CORS header。
   - 对有 `Origin` 的浏览器请求不返回 `Access-Control-Allow-Origin`。
4. 若命中 allowlist，返回对应 origin。
5. Allow-Headers 补齐：
   - `Origin`
   - `Content-Type`
   - `Authorization`
   - `X-Request-Id`
   - `X-Session-Token`
   - `x-api-key`
   - `anthropic-version`
6. 可补充：`Access-Control-Max-Age`。

验收：

- 配置 allowlist 后，匹配 origin 可通过预检。
- 不匹配 origin 不返回 CORS 允许头。
- `x-api-key` 与 `X-Session-Token` 预检通过。

### 5.7 任务 G：Relay context / timeout 传播

涉及文件：

- `relay/relaycommon/relayinfo.go`
- `relay/relayer.go`
- `relay/adaptor/adaptor.go`
- `relay/adaptor/openai/adaptor.go`
- `relay/adaptor/anthropic/adaptor.go`
- `relay/adaptor/responses/adaptor.go`

改造步骤：

1. 在 `RelayInfo` 增加：

```go
Context context.Context
```

2. `handle()` 中创建请求上下文：

```go
ctx := c.Request.Context()
if r.cfg.RequestTimeout > 0 {
    var cancel context.CancelFunc
    ctx, cancel = context.WithTimeout(ctx, time.Duration(r.cfg.RequestTimeout)*time.Second)
    defer cancel()
}
info.Context = ctx
```

3. `SameChannelDelay` 使用 `info.Context`，不再直接使用 `c.Request.Context()`。
4. adaptor 接口保持 `DoRequest(info, body)` 不变，但内部改为：

```go
req, err := http.NewRequestWithContext(info.Context, http.MethodPost, url, body)
```

5. 删除 `var _ = context.Background` 占位。

兼容策略：

- `request_timeout=0` 保持原行为，不主动超时。
- 第一阶段使用统一 timeout 作用于完整请求生命周期。
- 后续阶段再拆分：
  - non-streaming timeout
  - streaming first-byte timeout
  - streaming idle timeout

验收：

- 客户端取消后，上游请求会取消。
- `request_timeout` 设置后，超过时间返回 timeout 类错误。
- `adaptor.HTTPClient().Timeout` 仍可为 0，由 context 控制。

### 5.8 任务 H：客户端取消与 timeout 错误分类

涉及文件：

- `relay/relayer.go`
- 可选新增：`relay/errors.go`

改造步骤：

1. 新增错误分类函数：

```go
func classifyRelayError(ctx context.Context, err error) RelayErrorCategory
```

建议分类：

```go
type RelayErrorCategory string

const (
    ErrorCategoryClientCanceled RelayErrorCategory = "client_canceled"
    ErrorCategoryTimeout        RelayErrorCategory = "timeout"
    ErrorCategoryUpstream       RelayErrorCategory = "upstream_error"
    ErrorCategoryInternal       RelayErrorCategory = "internal_error"
)
```

2. `doOnce()` 中：
   - 如果 `info.Context.Err() == context.Canceled`，返回 client canceled 类型。
   - 如果 `info.Context.Err() == context.DeadlineExceeded`，返回 timeout 类型。
3. `handleStream()` 中：
   - `writer.WriteChunk` 导致的 broken pipe/client canceled 不计入渠道失败。
   - 如果流已开始，仍然结束响应，但日志状态标记为 cancelled。
4. `relayWithFailover()` 中：
   - client canceled 直接返回，不调用 `RecordFailure`。
   - timeout 可以计入失败并 retry/failover。
   - upstream 5xx/网络错误按现有策略。

第一阶段可以先用最小实现：

- `context.Canceled` 不熔断。
- `context.DeadlineExceeded` 作为 504 retryable。
- 常见 broken pipe 通过错误字符串兜底识别。

验收：

- 取消客户端请求不会增加熔断失败计数。
- 上游超时会进入 retry/failover。
- 日志能区分 cancelled 与 upstream error。

### 7.2 集成测试 / 手工验证

1. 首启无密码：

```bash
rm -f apirelay.db
APIRELAY_SESSION_SECRET=secret go run .
```

预期：启动失败，提示设置 `APIRELAY_INITIAL_ADMIN_PASSWORD`。

2. 首启安全配置：

```bash
APIRELAY_INITIAL_ADMIN_PASSWORD='strong-password' \
APIRELAY_SESSION_SECRET='long-random-secret' \
go run .
```

预期：管理员创建成功，日志不出现完整密码。

3. insecure dev bootstrap：

```bash
APIRELAY_ALLOW_INSECURE_DEFAULT_ADMIN=true \
APIRELAY_SESSION_SECRET='dev-secret' \
go run .
```

预期：允许旧密码，但有 warning。

4. 大 body：

- 向 `/v1/chat/completions` 发送超过 relay limit 的 body。
- 向 `/api/auth/login` 发送超过 admin limit 的 body。

预期：返回 413。

5. timeout：

- 配置 `relay.request_timeout: 1`。
- 上游 test server sleep 3 秒。

预期：请求在约 1 秒超时，进入 retry/failover 或返回 504。

6. client cancel：

- 发起请求后主动断开客户端。

预期：上游请求 context canceled，渠道熔断失败计数不增加。

7. 前端：

```bash

npm run build
```

预期：生产构建登录页不显示 `admin / admin123`。

## 9. 风险与回滚

### 9.1 兼容性风险

- 首启继续兼容“自动 admin/admin123”，降低本地体验变更；生产若未关闭 insecure 模式仍有弱口令风险。
- 缓解：默认写 warning，README 与配置示例明确建议生产设置强密码、固定 session secret 并关闭 `allow_insecure_default_admin`。

### 9.2 Session 兼容风险

- 改为 HMAC session 后，旧 session 全部失效。
- 缓解：这是安全升级可接受行为；发布说明中提示重新登录。

### 9.3 Timeout 行为变化

- 如果用户设置了 `request_timeout`，长流式请求可能被整体 timeout 截断。
- 缓解：默认仍为 0；文档说明第一阶段 timeout 是完整请求级，后续阶段会拆分 stream first-byte/idle timeout。

### 9.4 CORS 行为变化

- 原先依赖任意 origin 的浏览器调用会失败。
- 缓解：配置 `server.cors_allowed_origins`，本地开发可设置 `*`。
