# APIRelay 数据转发问题根因分析报告

## 问题现象

用户报告通过 APIRelay 转发后出现：
1. **对话不连续** - 上下文丢失，模型无法理解之前的对话
2. **无法调用 Skill** - 工具调用（function calling）失败
3. **上下文缺失** - 多轮对话时模型回复与前文无关

## 对比分析：成熟项目 vs APIRelay

### One-API / New-API 的核心设计

**流式处理（关键）**：
```go
// one-api/relay/adaptor/openai/main.go
func StreamHandler(c *gin.Context, resp *http.Response, relayMode int) {
    scanner := bufio.NewScanner(resp.Body)
    scanner.Split(bufio.ScanLines)
    
    common.SetEventStreamHeaders(c)
    
    for scanner.Scan() {
        data := scanner.Text()
        // 1. 直接透传原始行，不做任何修改
        if 正常SSE行 {
            render.StringData(c, data)  // 原样写回客户端
        }
        
        // 2. 仅做统计：解析 JSON 提取 usage/content
        var streamResponse ChatCompletionsStreamResponse
        json.Unmarshal([]byte(data[dataPrefixLength:]), &streamResponse)
        responseText += choice.Delta.Content
        if streamResponse.Usage != nil {
            usage = streamResponse.Usage
        }
    }
    
    if !doneRendered {
        render.Done(c)
    }
}
```

**关键特性**：
1. **零改写原则** - 上游返回什么就转发什么（除非明确需要协议转换）
2. **逐行透传** - 每扫描到一行 SSE 立即 `render.StringData(c, data)` 写回客户端
3. **元数据解析与业务分离** - 解析 JSON 只为统计 usage，不影响转发
4. **[DONE] 保真** - 上游发的 `data: [DONE]` 原样转发
5. **错误容忍** - 无法解析的行直接透传，不中断流

### APIRelay 的当前实现（存在问题）

**流式处理**：
```go
// relay/adaptor/openai/adaptor.go
func (a *Adaptor) StreamHandler(info, resp, onChunk) (*dto.Usage, error) {
    scanner := bufio.NewScanner(resp.Body)
    var usage *dto.Usage
    
    for scanner.Scan() {
        line := strings.TrimSpace(scanner.Text())
        if line == "" || !strings.HasPrefix(line, "data:") {
            continue  // ❌ 跳过空行
        }
        data := strings.TrimSpace(strings.TrimPrefix(line, "data:"))
        if data == "[DONE]" {
            break  // ❌ 直接跳出，不转发 [DONE]
        }
        
        chunk, err := apicompat.ParseOpenAIStreamChunk([]byte(data))
        if err != nil {
            continue  // ❌ 解析失败直接跳过，不透传原始行
        }
        
        if chunk.Usage != nil {
            usage = chunk.Usage
        }
        if err := onChunk(chunk); err != nil {  // ❌ 转换为 IR 后写
            return usage, err
        }
    }
    return usage, nil
}
```

**问题点**：
1. **强制解析** - 必须成功解析为 IR，失败则跳过该 chunk
2. **数据重构** - `ParseOpenAIStreamChunk` → IR → `WriteChunk` → 对外协议，经过多次序列化/反序列化
3. **[DONE] 丢失** - `break` 直接跳出循环，不向下游发送 `data: [DONE]`
4. **空行过滤** - SSE 协议要求的空行心跳被丢弃
5. **错误不透传** - 上游发送的特殊事件（如工具调用相关）若解析失败，客户端收不到

## 根本原因

### 1. 过度抽象的中间表示（IR）

APIRelay 引入了 `dto.UnifiedRequest` / `dto.UnifiedStreamChunk`，试图统一所有协议：

```go
type UnifiedStreamChunk struct {
    Delta   *Delta  `json:"delta,omitempty"`
    Usage   *Usage  `json:"usage,omitempty"`
    // ...
}
```

**问题**：
- **信息丢失** - OpenAI 的 `tool_calls`、Anthropic 的 `content_block` 等特殊字段在 IR 转换时丢失
- **解析失败导致数据丢弃** - 任何无法映射到 IR 的 chunk 都会被 `continue` 跳过
- **重构开销** - 每个 chunk 都经过：`原始bytes → OpenAI结构 → IR → 对外协议bytes`

### 2. 流式协议不完整

**缺失的关键元素**：
- ❌ 没有转发 `data: [DONE]` - 许多客户端依赖此标记判断流结束
- ❌ 没有保持 SSE 心跳（空行）- 可能导致超时或连接被中间代理切断
- ❌ 没有透传 `event:` 类型（如 OpenAI 的 `event: content_block_start`）

**CC Switch 的教训**：
根据 `.narrafork/cc-switch` 的文档，它们遇到的核心问题是：
- **跨后端 encrypted_content 无法解密** - 续聊时上游无法解析前一个后端生成的密文
- **model_provider 标签隔离** - 会话历史按供应商标签分抽屉，切换后看不到

这说明：**任何修改或重构上下文相关字段都可能导致续聊失败**。

### 3. 工具调用（Function Calling）数据被破坏

OpenAI 的工具调用流：
```
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"id":"call_xxx","type":"function","function":{"name":"get_weather","arguments":""}}]}}]}
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"function":{"arguments":"{"}}]}}]}
data: {"choices":[{"delta":{"tool_calls":[{"index":0,"function":{"arguments":"location"}}]}}]}
...
```

**APIRelay 的问题**：
- `dto.UnifiedStreamChunk` 没有 `ToolCalls` 字段
- `apicompat.ParseOpenAIStreamChunk` 无法解析 `tool_calls`
- 客户端收到的流中工具调用信息缺失 → Skill 调用失败

## 解决方案

### 方案A：零改写透传（推荐，最小改动）

**核心思路**：仅在协议确实不同时才转换，相同协议直接透传原始 bytes。

```go
// relay/adaptor/openai/adaptor.go
func (a *Adaptor) StreamHandler(info, resp, onChunk) (*dto.Usage, error) {
    scanner := bufio.NewScanner(resp.Body)
    var usage *dto.Usage
    var responseText strings.Builder
    
    for scanner.Scan() {
        line := scanner.Text()
        
        // 1. 直接透传原始行（保留 SSE 格式）
        if err := onChunk(&dto.RawChunk{Raw: line}); err != nil {
            return usage, err
        }
        
        // 2. 仅做统计（不影响转发）
        if strings.HasPrefix(line, "data:") {
            data := strings.TrimSpace(strings.TrimPrefix(line, "data:"))
            if data == "[DONE]" {
                continue
            }
            
            var chunk dto.OpenAIStreamChunk
            if err := json.Unmarshal([]byte(data), &chunk); err == nil {
                for _, choice := range chunk.Choices {
                    if choice.Delta.Content != nil {
                        responseText.WriteString(*choice.Delta.Content)
                    }
                }
                if chunk.Usage != nil {
                    usage = chunk.Usage
                }
            }
            // 解析失败也不影响转发，因为已经在步骤1透传了
        }
    }
    
    return usage, scanner.Err()
}
```

**修改点**：
1. `dto.UnifiedStreamChunk` 改为 `dto.RawChunk{Raw string}`
2. `Outbound.WriteChunk` 直接写原始 SSE 行，不做任何序列化
3. 仅在**对外协议与上游协议不同**时才启用 IR 转换（如 OpenAI → Anthropic）

### 方案B：完善 IR 支持工具调用（工作量大）

扩展 `dto.UnifiedStreamChunk` 支持所有 OpenAI 字段：
```go
type UnifiedStreamChunk struct {
    Delta     *Delta      `json:"delta,omitempty"`
    ToolCalls []ToolCall  `json:"tool_calls,omitempty"`  // 新增
    Usage     *Usage      `json:"usage,omitempty"`
    // ... 其他扩展字段
}
```

**问题**：每次 OpenAI/Anthropic 协议更新都需要同步修改 IR，维护成本高。

## 实施状态

### ✅ 已完成（v0.2.0）

1. **RawChunk 零改写透传模式** - 同协议直接透传原始 SSE 行
2. **[DONE] 标记修复** - 流结束时正确发送终止标记
3. **SSE 心跳保留** - 空行与 event: 行完整透传
4. **协议自适应** - 同协议零改写，跨协议走 IR 转换

**核心改动**：
- `dto.UnifiedStreamChunk` 新增 `Raw string` 字段
- `OpenAI Adaptor` 检测 `EndpointType` 决定是否启用 Raw 模式
- `Outbound` 优先写出 Raw，回退到 IR 序列化

**效果**：
- 工具调用(function calling)完整保留
- 特殊事件（content_block_start 等）不再丢失
- 上下文连续性恢复
- 性能提升（避免多次序列化/反序列化）

---

**结论**：APIRelay 的核心问题不是"转发失败"，而是"过度处理"——试图解析和重构所有数据导致信息丢失。成熟的网关（one-api）采用**透传优先、按需转换**的策略，这是保证上下文完整性的关键。
