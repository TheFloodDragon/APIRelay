# APIRelay 聚合站重构 + UI 重设计方案

## 一、目标

1. 把渠道（供应商）下的「模型」从逗号字符串升级为对象列表，每个模型可：启用/禁用、协议覆盖、上游名映射。
2. 协议解析支持：模型显式协议、供应商正则规则、供应商默认协议、全局正则规则。
3. 新增「模型聚合」全局视图，按显示名聚合（允许跨供应商重复）。
4. 全站 UI 重设计，支持亮/暗双主题，更灵动美观。

---

## 二、后端改动

### 2.1 数据模型（model 包）

**Channel 新增字段**（`model/channel.go`，AutoMigrate 自动加列，旧行留空兼容）：

```go
ModelConfigs  string // JSON: []ChannelModel，每模型的启用/协议/映射
ProtocolRules string // JSON: []ProtocolRule，供应商级正则规则
```

保留 `Type`（供应商默认协议 + 探测/鉴权用）、`Models`（兼容旧数据与回填）、`ModelMapping`（兼容）。

新增结构：
```go
type ChannelModel struct {
    Name     string `json:"name"`     // 显示名（对外请求名，Ability 索引键）
    Enabled  bool   `json:"enabled"`
    Protocol string `json:"protocol"` // "" | openai | anthropic | responses
    Upstream string `json:"upstream"` // "" 表示与 Name 相同
}
type ProtocolRule struct {
    Pattern  string `json:"pattern"`  // 正则
    Protocol string `json:"protocol"` // openai | anthropic | responses
}
```

Channel 新方法：
- `ModelConfigList()`：解析 ModelConfigs；为空时从旧 `Models` 派生（全部启用、无覆盖）保证向后兼容。
- `EnabledModelConfigs()` / `EnabledModelNames()`：仅启用项，用于 Ability 同步与聚合。
- `ProtocolRuleList()`：解析 ProtocolRules。
- `MappedModel(display)` 增强：优先用 ModelConfigs 里的 `Upstream`，回退旧 `ModelMapping`，再回退原名。
- `CreateChannel/UpdateChannel`：保存前把启用模型名回填到 `Models`，使旧聚合查询继续有效。

**Ability 同步**（`model/ability.go`）：`SyncChannelAbilities` 改为遍历 `EnabledModelConfigs()`，仅启用模型建索引（禁用模型不参与路由）。保留通配 `*`。

**新增 Setting 表**（`model/setting.go`）：键值存储全局配置。
```go
type Setting struct {
    Key   string `gorm:"primaryKey;size:64"`
    Value string `gorm:"type:text"`
}
GetSetting(key) / SetSetting(key,val)
```
全局协议规则存于 key `protocol_rules`（JSON `[]ProtocolRule`）。
提供 `GetProtocolRulesCached()`：内存缓存 + `SetSetting` 时失效，避免每请求查库。

`model/db.go` migrate 增加 `&Setting{}`。

### 2.2 协议常量（constant 包）

`constant/api_type.go` 新增：
```go
func APITypeFromName(name string) (APIType, bool) // "openai"/"anthropic"/"responses"
func APITypeName(t APIType) string
```

### 2.3 协议解析（relay 包）

新增 `relay/protocol.go`：
```go
func ResolveAPIType(ch *model.Channel, displayModel string) constant.APIType
```
解析优先级（**说明见下方调整点**）：
1. 模型显式协议（ChannelModel.Protocol）
2. 供应商正则规则命中
3. 全局正则规则命中
4. 供应商默认协议（Channel.Type）

内置正则编译缓存（`map[string]*regexp.Regexp` + RWMutex）。

> **优先级调整说明**：你在选项里选择的字面顺序是「供应商默认协议 > 全局正则」。但供应商默认协议（Type）对每个渠道都恒有值，若排在全局正则之前，全局正则将永远不会触发、形同虚设。因此我把「供应商默认协议」作为最末兜底（它是最不具体、覆盖面最广的规则），让全局正则真正可用。其余仍遵循「越具体越优先（局部优先）」：模型显式 > 供应商正则 > 全局正则 > 供应商默认。若你坚持原字面顺序，我可改回（但全局正则将不生效）。

`relay/relayer.go` 修改：
- `info.ApiType = ResolveAPIType(ch, info.OriginModel)`（替换 `ch.APIType()`）。
- `GetAdaptor(info.ApiType)` 不变。
- `channelTypeOf`/日志 ApiType 改用解析后的协议名。

### 2.4 控制器与路由（controller / router）

- `controller/channel.go`：Create/Update 已用 ShouldBindJSON 整体绑定，新增字段自动接收；补充默认值处理。
- 新增 `controller/model.go`：
  - `GET /api/models`：聚合所有渠道启用模型，按显示名分组，返回 `{name, providers:[{channel_id,channel_name,protocol,upstream,enabled}]}`。
  - `PUT /api/channels/:id/models/:name`：快捷启用/禁用单个模型（可选，UI 也可走整体 Update）。
- 新增 `controller/setting.go`：
  - `GET /api/settings/protocol-rules`、`PUT /api/settings/protocol-rules`（全局正则规则）。
- 新增 `GET /api/protocols`：返回可选协议列表 `[{value,name}]` 供前端下拉。
- `router/router.go` 注册以上路由。

## 三、前端改动（web/src）

### 3.1 主题系统（亮/暗）

- `tailwind.config.js`：`darkMode: 'class'`，扩展暗色友好的语义色板（surface/border/text 等用 CSS 变量或直接 dark: 变体）。
- `style.css`：所有组件类（btn/input/card/table/badge/modal…）补 `dark:` 变体；新增 CSS 变量主题。
- 新增 `composables/useTheme.js`：读写 localStorage、切换 `<html>.dark`、跟随系统。
- App.vue 顶栏加主题切换按钮。

### 3.2 整体视觉升级

- App.vue 外壳：更现代的侧边栏（图标+标签、active 高亮渐变、收拢动效）、顶栏（面包屑、主题切换、用户菜单）。
- 统一动效：卡片 hover、模态 fade/slide、列表 stagger、骨架屏。
- 配色更灵动（保留 brand 紫但增加渐变/玻璃拟态点缀），圆角与阴影统一。

### 3.3 各页面

- **Login.vue**：重设计登录卡片（渐变背景、品牌区）。
- **Dashboard.vue**：统计卡片重做（图标、趋势、渐变描边），保持数据逻辑。
- **Channels.vue → Providers（供应商）**：
  - 列表展示供应商（名称/默认协议/分组/模型数/状态）。
  - 编辑弹窗重做：基本信息 + 「模型管理」表格（每行：显示名、启用开关、协议覆盖下拉、上游名输入）+ 「协议规则」编辑器（正则+协议，可增删）+ 高级配置。
  - 拉取模型后并入模型表格（默认启用、协议=继承）。
- **新增 Models.vue（模型聚合）**：全局模型列表，按显示名聚合，展示提供它的供应商与各自协议；可快捷启用/禁用。
- **新增 Settings.vue（或并入某页）**：全局协议正则规则管理。
- **Tokens.vue / Logs.vue**：套用新设计系统重做模板，逻辑不变。
- `router.js` / 侧边栏 nav 增加「模型」「设置」入口。
- `api.js` 不变（已通用）。

### 3.4 构建

改完前端后运行 `npm run build` 重新生成 `web/dist`，供 Go embed。

## 四、实施顺序

1. 后端：constant → model（Channel/Ability/Setting）→ relay 协议解析 → relayer 接入 → controller/router。
2. `go build` + `go vet` + 现有测试通过；补充协议解析与模型配置的单元测试。
3. 前端：主题系统 → 设计系统(style.css) → App 外壳 → 各页面 → 新增页面。
4. `npm run build` 生成 dist。
5. 整体自测（构建通过、关键接口 mock 验证）。

---

## 五、兼容性与风险

- 旧渠道无 ModelConfigs：通过派生逻辑继续工作，UI 编辑后自动升级为对象列表。
- Ability 索引键仍是显示名，`/v1/models` 聚合行为不变。
- 协议解析优先级调整点已在 2.3 标注，需你确认。
- 前端为较大改动，分页面提交，确保每步可构建。
